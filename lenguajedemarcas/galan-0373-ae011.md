# PART 2 - Cas de decisió raonada

## CAS GRIS 1:

```
{
  "id": "1025",
  "usuari": "pau",
  "actiu": true
}
```

Justificació (6-8 línies):

En aquest cas, com a sistema, deixaria passar aquesta informació amb una advertència.

Deixaria passar perquè en entorns web, apis (on més informació amb aquest tipus de format hi ha), és comú que estiguin les dades en format tipus text, com podem veure a l'id. Fer la conversió de text a enter evita errors innecessaris.

També es podria aturar depenent del tipus de sistema que hi hagués, si és un sistema tipus banc, o metge, podria causar greus problemes a la base de dades, per això rebutjar-ho seria la millor opció.


## CAS GRIS 2:

```
{
  "id": 1028,
  "usuari": "roger"
}
```

Justificació (6-8 línies):

En aquest cas, com a sistema, deixaria passar aquesta informació amb una advertència.

Deixaria passar per garantir la continuïtat del servei, si el camp no és una clau primària, no hauria d'haver cap error amb assignar una clau automàtica, que en aquest cas seria "true/false". Això faria que per exemple: en cas que hi fos una aplicació antiga, als clients s'assigni un valor automàticament i que puguin continuar utilitzant l'aplicació sense errors.

També es podria rebutjar la sol·licitud, en el cas que el sistema s'inventi el valor que falta, podria anar en contra del que l'usuari volia fer. A l'hora de rebutjar la sol·licitud, es garanteix que no hi hagin dades a mitges que puguin causar errors a la base de dades posteriorment.


# PART 3

Per què validar dades no és suficient per decidir què fa un sistema?

R: Validar les dades només garanteix que la informació compleixi amb el format requerit, però no si té sentit o no dins del context del negoci/aplicació que vulgui el client.

S'hauria de fer un sistema capaç de gestionar els errors humans creant regles per gestionar, i que pugui prendre decisions, prioritzant la flexibilitat o la seguretat de les dades.
